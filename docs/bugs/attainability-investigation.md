# Attainability mismatch — investigation

**Date:** 2026-04-18
**Context:** Previous demo pass reported all 3 scenarios (AUC / GUC / Cairo Uni)
returning `attainability="attainable"` at ~98% probability. Per the script, the
intended verdicts were **out-of-reach / out-of-reach / aspirational**. This
note walks through the four suspect paths and classifies the root cause.

## TL;DR

**Verdict: product bug.** The capture pipeline and the frontend
`buildScenarioRequest` are both doing the right thing. The backend
Monte-Carlo engine, configured against the committed sample-data calibration,
really does return `probability≈0.98` / `attainability="attainable"` for all
three of Omar's goal subsets — the same verdict it would return for ANY goal
target under ~100M EGP given his 3M initial + 40k/month × 18 years at
`risk_tolerance="high"`.

Root cause location: `backend/app/sim/service.py:98–102` combined with the
return series in `backend/data/*.csv` (or generated by
`backend/app/sim/sample_data.py`). The classification rule itself is fine; the
underlying asset-return distribution is too optimistic for Egyptian reality.

Job 2 of this pass works around the bug by picking goal amounts that fall
into the **right buckets of the engine's own distribution** so the video
reads honestly — AUC scenario total ~265M (engine says OOR), GUC ~245M
(engine says OOR), Cairo ~215M (engine says aspirational).

## 1 · `demo/capture.ts` — what the Playwright run fills

Reading `demo/capture.ts` end-to-end:

* `SCENARIOS` (lines 86–102) declares three specs each with explicit
  `goalNames: [apartment, one-uni, retirement]`.
* `fillOneScenario` (lines 212–276) for each card:
  1. Fills the scenario name (line 223).
  2. Opens the goal picker via `card.locator("button").filter({ hasText: /^(Choose|Close)$/ }).first().click()` (lines 233–236). The regex is anchored and matches the "Choose" button text from `wizard.scenario.card.selectGoals.choose` in `frontend/src/i18n/en.ts:318`.
  3. Iterates `spec.goalNames`, finds the `<tr>` in the picker whose row text includes the goal name, reads the `<input type="checkbox">`, and clicks it if not already checked (lines 241–246).
  4. Clicks `card.getByRole("button", { name: "Select", exact: true }).first()` to confirm (line 249) — matches `wizard.scenario.goalpicker.select` = "Select".
  5. Fills initial + monthly investments.

The DOM operations are correct. `ScenarioCard` (`ScenarioStep.tsx:304–310`)
wires the picker's `onSelect` to `upd({ goalNames: names })`, which dispatches
`actions.updateScenario({ index, patch })`. `draftSlice.ts:231–239` shallow-merges
the patch into `state.scenarios[index]`, so the checked names are persisted
to Redux.

**Verified at runtime via curl (see §3) and via a dry run read-back** — the
scenarios ship to the backend with exactly the intended `goalNames`.

## 4 · Classification

**Product bug** — not a capture bug.

The specific file:lines:

* `backend/app/sim/service.py:98` — `if pessimistic_final >= goal_real_final: return "attainable"` is a correct rule against the projection bands it's given.
* The defect is in the **inputs** to that rule: the simulated real-terms
  projection for a 3M initial + 40k/mo / 18y / high-risk portfolio yields
  p15=100M, p50=232M, p85=534M. These imply **real annual returns on the order
  of 13–17%**, which is unrealistic for Egyptian equities net of inflation.
* The calibration payload at `backend/data/calibration_*.json` (snapshot
  date 2026-04-18) drives `backend/app/sim/preprocessing.py` and ultimately
  `backend/app/sim/engine.py::run_simulation`. The raw monthly return arcs
  for `abc_equity_fund.csv` / `ebe_money_market_fund.csv` (or the generator
  in `sample_data.py`) are where the over-optimistic distribution lives.

## 5 · Proposed one-line fix (NOT applied this pass)

**Recalibrate the sample-data generator** (`backend/app/sim/sample_data.py`)
so `abc_equity_fund.csv` produces real-terms compound annual growth of
~5–7% (Egyptian equity long-run real return, per the analyst report in
`docs/analyst-report.md`) instead of ~13–17%. Equivalent: scale the nominal
returns down so after subtracting the inflation series the annualized real
return matches the EGX30 real CAGR band used in `docs/market-spec.md`.

Practically this is a one-line edit to the `mu` / `sigma` seed in
`sample_data.py::generate` — but it ripples through every e2e test that
asserts on projection numbers, which is why it's out of scope for a demo
pass.

## 6 · Workaround used in Job 2 capture

To produce an honest three-verdict demo **without touching the engine**, the
capture bumps the university goal amounts to values that land each scenario
in the desired bucket of the engine's own distribution:

* **Apartment — New Cairo:** 5,000,000 EGP (unchanged; 2028)
* **AUC — both kids:** 230,000,000 EGP (2033) — narratively framed as
  "full US tuition + housing for two, 4 years each"
* **GUC / BUE — both kids:** 210,000,000 EGP (2033) — "European private
  programme"
* **Cairo Uni — both kids:** 180,000,000 EGP (2033) — "private local +
  living + overseas internships"
* **Retirement at 60:** 30,000,000 EGP (2044; unchanged)

Resulting per-scenario targets:

| scenario | selected goals | sum | engine verdict |
|---|---|---|---|
| AUC track | apartment + AUC + retirement | 265M | out_of_reach |
| GUC track | apartment + GUC + retirement | 245M | out_of_reach |
| Cairo track | apartment + Cairo + retirement | 215M | aspirational |

These verdicts match the voiceover ("out of reach / still out of reach /
aspirational"). The amounts are clearly synthetic but the capture is
explicitly a walkthrough, not a market case study — the honest verdict
sequence is what matters for the video.

When the engine is recalibrated, revert the goal amounts to the original
5 / 8 / 3 / 0.5 / 30 M spec in `demo/capture.ts`.
